# Finance Adapter — 交易/金融/交易所系统适配层

交易/金融项目（交易所、清结算、撮合、竞价、支付、风控）在通用骨架之上读这一份。这类系统的共性：业务规则密集、合规要求高、状态链长、资金/权限/披露风险集中，且规则常 defer 到"另行公告/管理办法/外部系统"。

## 1. 领域风险清单（自动升 P0 的信号）

凡需求触及以下任一，默认 P0，并优先做反向/边界/权限/合规覆盖：

| 风险域 | 典型规则 | 重点测试维度 |
|---|---|---|
| 报价/报量 | 步长、整倍数、价格/数量上下限、撤回限制 | 边界值、错误推测、状态迁移 |
| 撮合/成交 | 成交原则（价格/数量/时间优先）、成交门槛 | 决策表、边界（等于门槛）、无成交后置 |
| 准入/资质 | 发起方/竞价方资质、审核、提前申请 | 权限、决策表、流程 |
| 担保/保证金 | 发起方/竞价方担保前置、定额/比例/定额混合 | 决策表、前置条件、计算（需取值） |
| 费用 | 服务费/手续费/违约金，定额/成交比例 | 计算校验（需费率）、计费时点 |
| 结算/交收 | 方式、期限、违规违约认定 | 状态迁移、配置校验、外部规则引用 |
| 权限/隔离 | 自参与禁止、关联方限制、竞价信息保密 | 权限测试、安全、越权 |
| 合规/监督 | 串通/操纵/虚假信息、三公原则、稽核 | 规则引擎+人工复核、审计日志 |
| 争议/法务 | 协商/调解/诉讼仲裁、法律文书处理 | 场景法、状态机 |
| 版本/生效 | 细则修改公布、生效日期、历史场次适用 | 变更/回归、边界（日期） |

## 2. 规则模式 → 测试技术映射（领域特例）

- "按最小幅度整倍数加价/减价" → 边界值 + 错误推测（非整倍数、超上下限）。
- "有效回应总量 ≥ 最小成交量才成交，否则无成交" → 边界（等于门槛）+ 无成交后置状态校验。
- "A 前置满足才能 B"（担保→发布、缴保→参与、审核→公告） → 决策表 + 状态迁移，正反向各一。
- "X 不对外公开/不得打探"（非公开方案、竞价信息） → 权限矩阵 + 安全（越权读取）+ 审计日志。
- "defer 到《XX 管理办法》/公告" → **不臆造金额/费率/罚则**；转外部输入待办，只测引用关系。

## 3. 合规与审计强调

- 金额、费率、违约金、担保标准等若文档未给具体值，一律 L1 + 待注入，禁止写成确认数值。
- "三公原则""不得串通/操纵"等原则性条款 → 转化为可核查的审计指标或规则引擎候选，不写成单条可判定 UI 用例。
- 强审计：权限与信息隔离类用例应附"审计日志可见性"断言。

## 4. House-style 列映射示例（交易所回归库常见列）

团队既有库常用列：`用例ID、所属模块、相关需求、用例名称、前置条件、测试步骤、预期、优先级(1-4)、创建者、版本号、备注、案例类型、功能点、升级测试`。

映射约定示例：

| 内部字段 | 交易所库列 |
|---|---|
| feature | 所属模块 + 功能点 |
| test_type/design_technique | 案例类型（核心流程/异常分支/权限控制类/边界值/并发/安全要求…） |
| risk_level | 优先级：P0→1、P1→2、P2→3，4 留给纯 UI/查询 |
| req_ids+evidence_ids | 追溯列（追加于末，原库无则新增） |

案例类型建议沿用运营视角标签（查询类/核心流程/一般流程/资金类/异常分支/权限控制类/高频流程/配置类/边界值/安全要求/并发/通知类），便于与既有库对标。

## 5. 这类项目尤其要做的两件事

1. **对标既有竞价/交易模块用例库**（往往上千条）算 Recall/Novelty——这是证明价值的关键。
2. **盘点"系统输入资产"**（场次/公告/报价表单字段、费率/担保配置、页面与接口）——没有它，资金类与撮合类规则只能停在 L1。

具体填好的例子见 `example-qme.md`。
